# 模型与脚本职责分工

## 核心原则

**Skill设计时必须明确区分"给模型的指导"和"给脚本的验证"。**

## 分工矩阵

| 任务类型 | 执行者 | 原因 |
|----------|--------|------|
| 内容生成 | 🤖 模型 | 需要理解和创造能力 |
| 写作风格判断 | 🤖 模型 | 需要语义理解 |
| 质量审阅（内容层面） | 🤖 模型 | 需要理解上下文和逻辑 |
| 字数统计 | 🐍 脚本 | 纯数值计算 |
| 数量检查（notes、images） | 🐍 脚本 | 纯数值计算 |
| 格式验证 | 🐍 脚本 | 规则匹配 |
| 流程控制（分步生成、断点续传） | 🐍 脚本 | 状态管理 |
| 文件操作（合并、保存） | 🐍 脚本 | IO操作 |

## 常见错误

### ❌ 错误1：把所有逻辑都写成Python脚本

```python
# 错误：试图用脚本判断写作风格
def check_write_style(content):
    tech_patterns = ["支持", "提供", "实现"]
    has_tech_style = any(pattern in content for pattern in tech_patterns)
    return has_tech_style  # 过于简化，无法判断上下文
```

### ❌ 错误2：让模型做数值验证

```
请检查以下内容是否满足要求：
- 总字数是否≥15000
- 截图数量是否≥50
- 注意事项是否≥30
```
（模型计算数值不可靠，应该用脚本）

### ✅ 正确：明确分工

```
┌─────────────────────────────────────────────────────────────┐
│  Skill 指导层（SKILL.md + prompts/*.md）                     │
│  - 定义质量标准（字数、风格、结构）                           │
│  - 提供prompt模板给模型                                      │
│  - 描述正确/错误写法示例                                     │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  模型执行层（LLM）                                           │
│  - 生成内容（遵循skill中的写作规范）                          │
│  - 审阅内容质量（检查风格、逻辑、完整性）                     │
│  - 生成摘要和上下文                                          │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  脚本验证层（Python）                                        │
│  - 检查硬性指标（字数、数量）                                 │
│  - 流程控制（进度保存、断点续传）                             │
│  - 文件操作（合并、生成文档）                                 │
└─────────────────────────────────────────────────────────────┘
```

## 设计检查清单

设计新skill时，检查以下问题：

1. [ ] 是否有任务应该由模型执行但写成了脚本？
2. [ ] 是否有任务应该由脚本执行但让模型来做？
3. [ ] prompt模板是否清晰描述了输出格式？
4. [ ] 脚本是否只做数值计算和规则匹配？
5. [ ] 是否有混杂的伪代码需要拆分？
